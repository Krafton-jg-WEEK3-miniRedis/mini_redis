# 협업 규칙

팀원 전체가 동일한 기준으로 브랜치를 만들고 작업 내용을 구분할 수 있도록 아래 규칙을 사용한다.

## 1. 작업 유형
- 작업 유형은 소문자만 사용한다.
- 작업 유형은 변경의 핵심 목적을 기준으로 선택한다.

| 작업 유형 | 의미 |
| --- | --- |
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서 수정 |
| `style` | 코드 포맷팅, 세미콜론 누락 등 코드 동작 변화가 없는 수정 |
| `refactor` | 코드 리팩토링 |
| `test` | 테스트 코드 추가 및 테스트 리팩토링 |
| `chore` | 의존성 수정, 설정 변경, 단순 유지보수 |
| `db` | 스키마 변경 사항 |

## 2. 브랜치 이름 규칙
- 브랜치 이름은 `작업유형/작업내용` 형식으로 작성한다.
- 작업 내용은 영어로 작성한다.
- 단어 사이는 하이픈(`-`)으로 연결한다.
- 예시: `feat/implement-google-login`

## 3. main 브랜치 작업 금지
- 기능 개발이나 추가 작업은 `main` 브랜치에서 직접 진행하지 않는다.
- 작업을 시작하기 전에 반드시 규칙에 맞는 새 브랜치를 생성하고 그 브랜치에서 작업한다.
- `main` 브랜치는 리뷰가 끝난 변경만 반영하는 기준 브랜치로 사용한다.

## 4. 작성 예시
- `feat/add-resp-cli`
- `fix/handle-expired-key`
- `docs/update-readme`
- `test/add-hash-table-tests`
- `chore/add-mongodb-compose`

## 5. 팀 공통 기준
- 브랜치를 만들기 전에 작업 목적을 먼저 정리한다.
- 한 브랜치에는 하나의 주제만 담는다.
- 작업 유형이 헷갈리면 가장 큰 변경 목적을 기준으로 선택한다.
