# 팀원 2 데모/인프라 작업 정리

이 문서는 2번 팀원이 담당한 Mini Redis 데모 환경과 인프라 작업을 정리한 README입니다.  
주요 내용은 Docker 기반 실행 환경 구성, 데모 시나리오 연결, 대용량 데이터 준비, 원격 서버 운영 과정에서의 트러블슈팅입니다.

## 해당 범위

기획서 기준 2번 팀원의 기본 역할은 다음과 같았습니다.

- Docker 기반 데모 환경 구성과 실행 흐름 정리
- Mini Redis와 MongoDB 비교 시연을 위한 인프라 세팅
- 대용량 데이터 준비와 시드 파이프라인 구성
- 원격 서버 배포 환경 확인과 접속 경로 점검
- 데모 관점에서 필요한 CLI 사용 흐름 확인

## 구현 결과

2번 팀원 영역은 "서버와 저장소 구현 결과를 실제로 실행 가능한 데모 환경에서 확인할 수 있게 만드는 것"에 초점이 있습니다.

- Mini Redis와 MongoDB를 비교 설명할 수 있는 데모 시나리오를 정리했습니다.
- 원격 서버에서 Mini Redis가 어떤 주소와 포트로 열리는지 확인하는 흐름을 잡았습니다.
- Docker 기반 실행을 전제로 서비스 구성을 점검하고, 각 컨테이너 역할을 분리해 설명할 수 있게 했습니다.
- 대용량 데이터 적재를 위한 시드 작업 방향을 정리했습니다.
- 접속 실패, 포트 충돌, 잘못된 경로 설정 같은 운영 이슈를 직접 확인하고 해결 절차를 정리했습니다.

## 데모/인프라 구성

발표와 시연 기준의 전체 구성은 아래와 같습니다.

```text
Demo Web / Client
        |
        +--> Mini Redis (TCP/RESP2)
        |
        +--> MongoDB
```

핵심 목적은 같은 요청을 두 경로로 비교해 보여주는 것입니다.

- MongoDB 직접 조회 경로
- Mini Redis 캐시 조회 경로

이 구조를 통해 "DB를 직접 조회하는 경우"와 "캐시 서버를 먼저 조회하는 경우"의 차이를 시연할 수 있도록 구성했습니다.

## Docker 환경 정리

데모 환경은 Docker 기준으로 생각하고 정리했습니다.

- 데모 웹은 사용자 요청을 받고 비교 시나리오를 실행하는 진입점입니다.
- Mini Redis는 TCP/RESP2 기반으로 직접 요청을 받는 캐시 서버입니다.
- MongoDB는 비교 대상이 되는 원본 데이터 저장소입니다.

운영 과정에서 중요했던 포인트는 다음과 같습니다.

- 서비스별 포트가 겹치지 않도록 확인해야 합니다.
- `127.0.0.1` 바인딩과 `0.0.0.0` 바인딩의 차이를 구분해야 합니다.
- 같은 6379 포트가 열려 있어도, 실제 응답 주체가 Docker 컨테이너인지 직접 실행한 프로세스인지 확인이 필요합니다.
- 컨테이너 재시작만으로 반영되지 않는 변경은 이미지 재빌드 여부와 볼륨 마운트 여부를 같이 확인해야 합니다.

## 대용량 데이터 파이프라인

MongoDB와 Mini Redis 비교 시연을 위해서는 어느 정도 규모가 있는 데이터셋이 필요했습니다.  
그래서 단순 샘플 몇 건이 아니라, 반복 조회와 캐시 hit 여부를 설명할 수 있는 형태의 시드 데이터 준비 흐름을 잡았습니다.

- listing_id, category, location, title, price 같은 필드를 가진 데이터 구조를 기준으로 삼았습니다.
- 대량 insert 작업은 한 번에 몰아넣기보다 배치 단위로 나누는 방식을 고려했습니다.
- 시드 완료 후에는 바로 조회 테스트와 캐시 적중 테스트로 이어질 수 있게 시나리오를 연결했습니다.

## 트러블슈팅

작업하면서 실제로 많이 부딪힌 문제는 아래와 같습니다.

### 포트 충돌과 실행 주체 구분

가장 자주 헷갈렸던 부분은 6379 포트가 이미 사용 중인 상태에서, 지금 접속된 대상이 내가 방금 실행한 Mini Redis인지, Docker 안에서 이미 떠 있던 Mini Redis인지 바로 구분하기 어려웠다는 점입니다.

- 포트가 열려 있다고 해서 원하는 서버가 떠 있는 것은 아니었습니다.
- 같은 계정, 같은 서버 환경에서는 더 헷갈리기 쉬웠습니다.
- 접속 성공 전에 `ss`, `ps`, Docker 컨테이너 상태를 같이 확인해야 했습니다.

즉 "서버가 켜져 있는지"보다 "어떤 서버가 그 포트를 점유하고 있는지"를 먼저 확인하는 습관이 중요했습니다.

### 경로/배포 환경 불일치

원격 서버, 로컬 작업 디렉터리, Docker 컨텍스트 경로가 서로 다를 수 있어서, 실행 경로가 어긋나면 바로 접속 실패나 빌드 실패로 이어졌습니다.

- 현재 작업 디렉터리와 패키지 실행 위치를 정확히 맞춰야 했습니다.
- 컨테이너 내부 경로와 호스트 경로를 혼동하면 실행 명령이 실패했습니다.
- `python -m mini_redis.cli` 같은 명령은 패키지 루트에서 실행해야 정상 동작했습니다.

### 네트워크/방화벽 설정 점검

원격 서버에서 외부 접근을 테스트할 때는 코드보다 네트워크 설정이 먼저 문제가 되는 경우가 있었습니다.

- 서버가 `0.0.0.0`으로 바인딩되어 있는지 확인해야 했습니다.
- 방화벽이나 보안 규칙 때문에 외부에서 접근이 막히는 경우를 따로 확인해야 했습니다.
- 로컬 접속 성공과 외부 접속 성공은 전혀 다른 문제라는 점을 계속 점검했습니다.

## 작업 관점에서의 정리

2번 팀원 영역은 기능 구현을 직접 추가하는 작업이라기보다, 팀원들이 만든 결과물을 하나의 실행 가능한 시연 환경으로 묶는 역할에 가깝습니다.

- 3번 팀원이 만든 TCP 서버와 라우터가 실제 환경에서 붙는지 확인
- 4번 팀원이 만든 저장소 동작이 데모 요청 흐름에서 보이는지 확인
- 발표 시나리오에서 MongoDB direct와 Mini Redis cache 경로를 비교할 수 있게 구성
- 포트, 경로, 배포 환경, 접속 경로를 정리해 재현 가능성을 높임

## 변경 파일 기준 요약

2번 팀원 작업과 직접 관련된 산출물은 다음 범주로 정리할 수 있습니다.

- Docker 및 배포 환경 설정 파일
- 데모 웹 또는 비교 시연용 프런트엔드/백엔드 파일
- MongoDB 시드 스크립트와 데이터 적재 흐름
- 환경 변수 예시와 실행 방법 문서

현재 저장소에서는 위 산출물 일부가 직접 포함되어 있지 않더라도, 데모와 인프라 관점의 책임 범위는 위와 같이 정리할 수 있습니다.

## 정리

2번 팀원의 작업은 Mini Redis 프로젝트를 "코드가 있는 상태"에서 끝내지 않고, 실제로 실행하고 비교하고 발표할 수 있는 환경으로 연결하는 역할이었습니다.  
특히 Docker 환경, 포트 점유 상태, 원격 서버 접근, 대용량 데이터 준비, 실행 주체 구분 같은 운영 이슈를 직접 다루면서 인프라 관점에서 프로젝트를 정리했습니다.  
이 영역을 통해 팀 결과물이 단순 구현을 넘어, 재현 가능하고 설명 가능한 데모 형태로 이어질 수 있었습니다.
