# Mini Redis 팀 기획서

## Summary

- 목표: Mini Redis를 구현하고, MongoDB와의 성능 비교 데모 및 테스트 결과를 README 중심으로 발표한다.
- MVP 완료 기준: RESP2 TCP 서버, 커스텀 해시테이블 기반 SET/GET/DEL, redis-cli, 데모 웹, MongoDB 비교 벤치마크, 단위/통합 테스트.
- 기본 구조: Demo Web / redis-cli -> RESP2 서버(라우터) -> Hash Table 엔진, 데모 웹은 MongoDB 직접 조회와 Mini Redis 캐시 조회를 비교한다.

## Key Changes

- 1번 팀원: 데모 웹, QA, 테스트 발표 담당. MongoDB 직접 조회와 Mini Redis 캐시 조회의 응답시간을 비교하는 화면을 만들고, 테스트 결과와 발표 흐름을 README에 정리한다.
- 2번 팀원: redis-cli 담당. interactive 모드와 one-shot 모드를 제공하고, 명령을 RESP2로 인코딩해 서버에 전송한 뒤 응답을 디코딩해 터미널에 출력한다. 종료 명령은 CLI에서 EXIT를 지원한다.
- 3번 팀원: 서버/라우터 담당. TCP 연결, RESP2 파싱, 명령 디스패치, 에러 응답, PING/SET/GET/DEL/INFO/QUIT/EXIT 라우팅을 구현한다. 저장소 인터페이스는 set(key, value), get(key), delete(keys) -> int, expire(key, seconds)를 기준으로 잡는다.
- 4번 팀원: 저장소 엔진 담당. 별도 해시테이블 구조를 구현하고 SET/GET/DEL 핵심 로직을 맡는다. 1차 확장으로 EXPIRE를 위한 만료 시각 저장과 lazy expiration 체크 지점을 추가한다.

## Interfaces And Collaboration

- 서버 프로토콜은 RESP2 기준으로 유지하고, 종료는 QUIT와 EXIT를 모두 허용한다.
- 비교용 DB는 MongoDB로 고정하고, 데모 웹은 동일 키에 대해 MongoDB 직접 조회와 Mini Redis 경유 조회 두 경로를 제공한다.
- 의존성 표준은 pyproject.toml + docker-compose.yml + .env.example + 공통 실행 스크립트로 통일한다.
- 구현 순서는 저장소 인터페이스 확정 -> 해시테이블 구현 -> 서버 라우터 연결 -> CLI 연결 -> 데모 웹/벤치마크 -> README 정리로 고정한다.

## Test Plan

- unit: 해시 충돌, overwrite, 없는 키 조회, 여러 키 삭제 시 삭제 개수 반환, 만료 키 접근 시 삭제 및 miss 처리.
- integration: CLI에서 SET/GET/DEL/EXIT, 잘못된 RESP, 인자 수 오류, unknown command, 서버의 QUIT/EXIT 종료 처리.
- benchmark: 같은 데이터셋에 대해 MongoDB 직접 조회와 Mini Redis 캐시 조회를 반복 실행해 평균/최소/최대 응답시간을 비교한다.
- acceptance: README만으로 환경 실행, MongoDB 기동, 서버 실행, CLI 테스트, 데모 재현이 가능해야 한다.

## Assumptions

- 구현 언어는 현재 저장소 흐름에 맞춰 Python으로 유지한다.
- EXIT는 CLI와 서버 모두 지원하되, Redis 호환성을 위해 QUIT도 제거하지 않는다.
- 만료 정책은 백그라운드 청소보다 구현이 단순한 lazy expiration을 기본으로 한다.
- persistence는 MVP 필수 범위에서 제외하고, 필요 시 README에 확장 방향만 남긴다.
