# 팀원별 Codex 프롬프트

각 팀원은 아래 프롬프트를 그대로 복사해서 Codex에 입력하면 된다.

공통 규칙:
- `main` 브랜치에서는 직접 작업하지 않는다.
- 먼저 규칙에 맞는 작업 브랜치를 생성하고 이동한 뒤 작업한다.
- 작업 전 [기획서.md](./기획서.md)와 [협업규칙.md](./협업규칙.md)를 확인한다.

## 1번 팀원 프롬프트

```md
먼저 `main` 브랜치에서 직접 작업하지 말고, `feat/build-demo-benchmark-page` 브랜치를 생성해서 이동한 뒤 작업하세요.

`docs/기획서.md`를 기준으로 1번 팀원 역할을 구현하세요.

목표:
- Mini Redis와 MongoDB 성능 비교를 보여주는 데모 웹 페이지를 구현
- MongoDB 직접 조회와 Mini Redis 캐시 조회를 같은 데이터 기준으로 비교
- 평균/최소/최대 응답시간을 화면에 표시
- README에 데모 방법, 테스트 결과, 발표 흐름을 정리

요구사항:
- Python 기반으로 구현
- 데모 웹은 동일 키에 대해 `MongoDB 직접 조회`와 `Mini Redis 경유 조회` 두 경로를 제공
- QA/테스트 발표에 사용할 수 있게 benchmark 결과를 재현 가능하게 정리
- README만 보고 실행 가능한 수준으로 문서화
- 테스트나 실행 방법이 필요하면 공통 스크립트도 정리

작업 방식:
- 먼저 현재 저장소 구조를 분석
- 필요한 파일을 생성/수정
- 변경 후 실행 방법과 검증 결과를 정리
- 마지막에 어떤 파일을 바꿨는지와 남은 리스크를 요약
```

## 2번 팀원 프롬프트

```md
먼저 `main` 브랜치에서 직접 작업하지 말고, `feat/implement-redis-cli` 브랜치를 생성해서 이동한 뒤 작업하세요.

`docs/기획서.md`를 기준으로 2번 팀원 역할을 구현하세요.

목표:
- Mini Redis용 `redis-cli` 프로그램 구현
- interactive 모드와 one-shot 모드 둘 다 지원
- 사용자가 입력한 명령을 RESP2로 인코딩해 서버에 전송
- 서버 응답을 RESP 규격에 맞게 디코딩해서 터미널에 출력
- 종료 명령은 `EXIT`를 지원하고, 서버 호환성을 위해 `QUIT`도 처리 가능하게 고려

최소 지원 명령:
- `PING`
- `SET`
- `GET`
- `DEL`
- `EXIT`

요구사항:
- Python으로 구현
- interactive 모드에서는 여러 명령을 연속 입력 가능해야 함
- one-shot 모드에서는 실행 인자로 명령 1회를 보내고 종료
- bulk string, integer, error, array 응답을 구분해서 출력
- 잘못된 입력이나 연결 실패 시 에러 메시지를 명확히 출력
- 기본 사용법을 README 또는 관련 문서에 반영

작업 방식:
- 먼저 기존 서버 구현과 RESP 처리 방식을 분석
- CLI 파일 구조를 정리해서 구현
- 가능하면 간단한 테스트도 추가
- 마지막에 실행 예시와 검증 결과를 요약
```

## 3번 팀원 프롬프트

```md
먼저 `main` 브랜치에서 직접 작업하지 말고, `feat/build-resp-router-server` 브랜치를 생성해서 이동한 뒤 작업하세요.

`docs/기획서.md`를 기준으로 3번 팀원 역할을 구현하세요.

목표:
- TCP 기반 Mini Redis 서버/라우터 구현
- RESP2 요청 파싱, 명령 디스패치, 에러 응답 처리
- `PING/SET/GET/DEL/INFO/QUIT/EXIT` 라우팅 구현
- 저장소 엔진과 연결 가능한 인터페이스 계층 정리

저장소 인터페이스 기준:
- `set(key, value)`
- `get(key)`
- `delete(keys) -> int`
- `expire(key, seconds)`

요구사항:
- 현재 단일 파일 서버가 있으면 분석 후 역할별로 구조 분리
- RESP2 프로토콜 유지
- `QUIT`와 `EXIT` 모두 서버 종료 명령으로 지원
- 인자 수 오류, unknown command, 잘못된 RESP 입력에 대한 에러 응답 처리
- 4번 팀원이 만든 해시테이블 저장소와 쉽게 연결되도록 인터페이스 분리
- 테스트 가능하도록 단위 또는 통합 테스트 추가

작업 방식:
- 기존 `main.py`를 출발점으로 삼되 구조를 정리
- 프로토콜 처리, 라우팅, 응답 작성 책임을 분리
- 실행 방법과 테스트 방법을 문서화
- 마지막에 변경 파일, 테스트 결과, 남은 이슈를 요약
```

## 4번 팀원 프롬프트

```md
먼저 `main` 브랜치에서 직접 작업하지 말고, `feat/implement-hash-table-store` 브랜치를 생성해서 이동한 뒤 작업하세요.

`docs/기획서.md`를 기준으로 4번 팀원 역할을 구현하세요.

목표:
- Mini Redis의 핵심 저장소인 커스텀 해시테이블 구현
- `SET/GET/DEL` 핵심 로직 구현
- 1차 확장으로 `EXPIRE`를 위한 lazy expiration 지원

요구사항:
- Python 기본 `dict`를 직접 저장소로 쓰지 말고, 별도 해시테이블 구조를 구현
- 버킷, 충돌 처리, 삽입/조회/삭제 로직을 명확히 분리
- overwrite 동작 지원
- 여러 키 삭제 시 삭제된 개수를 정수로 반환
- `expire(key, seconds)` 형태로 만료 시간 저장
- lazy expiration 방식으로 조회/접근 시 만료 여부 확인 후 삭제 처리
- 단위 테스트 추가: 충돌, overwrite, miss, multi-delete, expired key 처리

작업 방식:
- 저장소 인터페이스를 먼저 명확히 설계
- 서버가 쉽게 붙일 수 있도록 독립 모듈로 구현
- 테스트 코드를 함께 작성
- 마지막에 자료구조 설명, 테스트 결과, 통합 시 주의점을 요약
```
